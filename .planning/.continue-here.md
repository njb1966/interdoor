<decisions_made>

- Quarantine, not delete, faulty Dominion code. Reason: preserve historical material for proper
  Phase 2 review while removing it from Phase 1 build/release/test surface.
- CLI-first hub administration. Reason: safer and simpler for the current sole-admin Phase 1
  model; no new network admin surface.
- Suspended nodes cannot authenticate. Reason: a compromised or faulty node should be stopped
  from syncing, not merely hidden from public directory output.
- Continuous test hardening. Reason: federation/security issues should be locked down as they are
  fixed, not batched later.
- Standard event validation at the hub. Reason: Phase 2 needs a stable event contract before adding
  another game.
- Legacy docs are quarantined with warnings, not rewritten wholesale. Reason: documentation is
  important, but extracting a generic game-node framework from one approved game would be premature.
- Queue failed/blocked lifecycle: transient resolver/import failures stay `pending`; malformed
  non-retryable queue payloads may be marked `blocked` only by the victim/destination owner and
  retried locally by an operator after repair.
- SQLite is accepted for the Phase 1 public hub while InterDoor remains a small network with
  local-only administration and low write volume. Production backups must use online backup or
  service-aware copy procedures, and restore/readback drills must stay part of hardening.

</decisions_made>
